You've invested in a keyboard-first workflow everywhere — Neovim for editing,
Yabai for [window management]({% post_url 2024-07-04-yabai %}), skhd for
shortcuts — and then you open a browser and your hand reaches for the mouse.
Clicks to switch tabs. Clicks to navigate history. Clicks to follow links. It's
all friction you've already solved everywhere else.

[Vimium](https://vimium.github.io) is a browser extension that brings Vim-style
keybindings to Chrome and Firefox. Pages become navigable without a mouse —
scroll with `j/k`, jump to links with `f`, switch tabs with `J/K`. It's the
missing piece in a keyboard-first setup.

## Default Keybindings Worth Knowing

Vimium ships with sensible defaults. A few you'll use constantly:

- `f` — show link hints; type the hint characters to follow the link
- `F` — same, but open in a new tab
- `j/k` — scroll down/up
- `gg/G` — top/bottom of page
- `/` — find on page (Vim-style)
- `gi` — jump focus to the first input field
- `o/O` — open Vomnibar (URL or search) in current/new tab
- `r` — reload
- `yy` — copy current URL

That last one — `yy` — is immediately useful. "Yank the URL" maps perfectly onto
the mental model.

## Custom Keybindings

The defaults are good, but a few of them clash with what I've internalized from
Vim and the rest of my setup. Here's what I've changed:

```text
map h previousTab
map l nextTab
map [ goBack
map ] goForward
map <c-d> scrollPageDown
map <c-u> scrollPageUp
map <c-f> scrollFullPageDown
map <c-b> scrollFullPageUp
map u restoreTab
map x removeTab
map b createTab
```

A few things worth noting here.

1. `h/l` for tab switching instead of Vimium's default `J/K`. Tab navigation is
   frequent enough that it should be on the home row, not shifted keys. It also
   maps naturally onto "move left/right."
2. `[/]` for back/forward in history. This keeps the same bracket convention I
   use for buffer navigation in Neovim — muscle memory transfers across tools.
3. `<c-d>/<c-u>/<c-f>/<c-b>` for scrolling. These are standard Vim page movement
   bindings and they feel immediately at home.
4. `x` to close a tab, `u` to restore it. `x` is cut/delete, `u` is undo — which
   is exactly what each does.

## Link Hints

When you press `f`, Vimium overlays every clickable element on the page with a
letter combination. You type the combination to follow it. The default hint
characters span the whole alphabet, which means you'll often need two or three
keystrokes per link. I've narrowed it to the home row:

```text
linkHintCharacters: asdfghjkl
```

Nine characters is plenty for most pages, and every hint is reachable without
moving your fingers. The one tradeoff: on dense pages with hundreds of links,
combinations get longer. In practice this almost never matters.

## Exclusion Rules

Not every site should be Vimium'd. Some web apps have their own keyboard
shortcuts that conflict — Vimium's `j/k` fights with Google Docs' line
navigation, and Jira has its own shortcut layer. I exclude both, plus one more
that should be obvious:

- `https://docs.google.com/*` — Google Docs handles its own keys
- `https://panther-labs.atlassian.net/*` — Jira does too
- `https://keyboard-test.space/*` — a keyboard testing site should receive raw
  key input, not browser extension intercepts

Vimium's settings let you add exclusion rules as URL glob patterns. Any matched
page gets Vimium entirely disabled — not just a few keys passed through, but the
whole extension stepped back.

One more setting worth enabling: **Grab back focus**. Some pages steal keyboard
focus when they load — ads, modals, embedded iframes. With this on, Vimium
reclaims focus back to the page so your keyboard shortcuts keep working.
